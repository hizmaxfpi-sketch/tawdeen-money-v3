تقرير تدقيق شامل - مشروع توطين (Tawdeen)
========================================

تاريخ التقرير: 2024-05-22
الدور: مطور Full-Stack وباحث أمني خبير

مقدمة:
تم إجراء تدقيق شامل لكود المصدر وهيكل المشروع (Vite + React + Supabase). يهدف هذا التقرير إلى تحديد نقاط الضعف البرمجية، الثغرات الأمنية، واقتراح خريطة طريق استراتيجية للتحسين.

---------------------------------------------------------
1. الثغرات الأمنية (Security Vulnerabilities)
---------------------------------------------------------

● [Critical] تجاوز الصلاحيات في وظيفة استعادة النسخ الاحتياطي (Edge Function):
  - المشكلة: وظيفة `restore-backup` تستخدم `service_role_key` للقيام بعمليات الـ Upsert، وتتحقق فقط من وجود جلسة مستخدم (authenticated) دون التأكد من رتبته (Admin/Owner). يمكن لأي مستخدم مسجل إرسال بيانات JSON وتعديل أو حذف بيانات في جداول النظام.
  - المنطق: استخدام مفتاح الخدمة يتجاوز سياسات أمن الجداول (RLS)، لذا يجب تقييد الوصول للوظيفة برمجياً.
  - الحل المقترح:
    ```typescript
    // داخل وظيفة Edge Function
    const { data: { user } } = await userClient.auth.getUser();
    const { data: roleData } = await supabase.from('platform_admins').select('id').eq('user_id', user.id).maybeSingle();
    if (!roleData) return new Response("Unauthorized", { status: 403 });
    ```

● [Medium] الاعتماد على التحقق من الصلاحيات في جهة العميل (Client-side):
  - المشكلة: تعتمد واجهة المستخدم على hooks مثل `useUserRole` لإخفاء الأزرار، بينما يجب أن تكون القيود صارمة في قاعدة البيانات عبر RLS و RPCs.
  - الحل المقترح: مراجعة كافة دوال RPC والتأكد من أنها تستخدم `auth.uid()` للتحقق من هوية المنفذ وصلاحياته قبل التعديل.

● [Low] تسريب ملفات البيئة (.env):
  - المشكلة: ملف `.env` مدرج في سجلات المشروع. رغم أن مفتاح Supabase هو "Publishable"، إلا أن كشفه في المستودع ممارسة غير آمنة.
  - الحل المقترح: إضافة `.env` إلى ملف `.gitignore` واستخدام "Secrets" في منصة الاستضافة.

---------------------------------------------------------
2. العيوب البرمجية والديون التقنية (Bugs & Technical Debt)
---------------------------------------------------------

● [High] تكرار مخازن الحالة (Redundant State Management):
  - المشكلة: وجود مخازن قديمة تعتمد على LocalStorage (`useFinanceStore`, `useShippingStore`) تزامناً مع استخدام Supabase. هذا يسبب تشتتاً في مصدر الحقيقة (Source of Truth).
  - الحل المقترح: حذف كافة الـ hooks التي تعتمد على التخزين المحلي وتوحيد التعامل عبر `useSupabaseFinance` و `useSupabaseShipping`.

● [Medium] جداول مهملة وقواعد بيانات غير موحدة:
  - المشكلة: جدول `ledger_accounts` موصوف بأنه "Deprecated" في الكود، لكنه لا يزال موجوداً ويُستخدم في بعض الدوال.
  - الحل المقترح: دمج كافة الحسابات في جدول `contacts` وتحديث العلاقات في جداول `transactions` و `debts` لحذف الاعتماد على الجداول القديمة.

● [Low] أداء جلب البيانات (Pagination):
  - المشكلة: يتم جلب حتى 1000 عملية مالية دفعة واحدة في `useTransactions.ts`. مع نمو البيانات، سيتأثر أداء التطبيق بشكل ملحوظ.
  - الحل المقترح: تنفيذ "التصفح اللانهائي" (Infinite Scrolling) أو الترقيم الفعلي باستخدام `limit` و `offset` برمجياً.

---------------------------------------------------------
3. توصيات واجهة وتجربة المستخدم (UI/UX Recommendations)
---------------------------------------------------------

● تحسين "التصميم المتنفس" (Breathable Design):
  - التوصية: زيادة المساحات البيضاء (Padding/Gap) في بطاقات لوحة التحكم (Dashboard cards) لتقليل العبء المعرفي.
  - المنطق: التطبيق مليء بالأرقام والبيانات المالية، المساحات تساعد العين على التركيز.

● توحيد الهوية البصرية للعمليات:
  - التوصية: استخدام أيقونات ملونة ومميزة لكل تصنيف مالي (مثلاً: أيقونة شاحنة لعمليات الشحن، حقيبة للمشاريع).
  - المنطق: يسهل على المستخدم تمويل العمليات بصرياً دون الحاجة لقراءة النص دائماً.

● واجهة سطح المكتب (Desktop Navigation):
  - التوصية: إضافة شريط جانبي (Sidebar) ثابت لنسخة سطح المكتب بدلاً من الاعتماد الكلي على الشريط السفلي.
  - المنطق: الاستفادة من عرض الشاشة لزيادة سرعة الانتقال بين الوحدات.

---------------------------------------------------------
4. الرؤية الإبداعية والتكامل (Creative Vision)
---------------------------------------------------------

● أتمتة المطابقة البنكية (AI Reconciliation):
  - الميزة: إمكانية رفع ملف كشف حساب بنكي (Excel/CSV) واستخدام الذكاء الاصطناعي لمطابقته مع العمليات المسجلة يدوياً واقتراح القيود المفقودة.

● نظام إشعارات ذكي عبر WhatsApp:
  - الميزة: إرسال تحديثات تلقائية للعملاء عند تغير حالة الحاوية أو تأكيد استلام الدفعات المالية مباشرة من النظام.

● مركز التحليلات التنبؤية (Predictive Analytics):
  - الميزة: بناء نظام تنبؤ بالتدفق النقدي (Cash Flow Forecasting) بناءً على تواريخ استحقاق الديون السابقة ومواعيد وصول الحاويات المتوقعة.

● التكامل العميق بين الشحن والمالية:
  - الميزة: ربط تكاليف الشحن (Customs, Port, etc.) تلقائياً بحسابات الموردين في النظام المالي لضمان عدم نسيان أي قيد محاسبي.

---------------------------------------------------------
خاتمة:
مشروع "توطين" يمتلك أساساً متيناً، ولكن يتطلب تركيزاً على تنظيف الديون التقنية وإغلاق الثغرات الأمنية في وظائف السحاب (Edge Functions) ليصبح نظاماً محاسبياً متكاملاً وجاهزاً للتوسع الكبير.
