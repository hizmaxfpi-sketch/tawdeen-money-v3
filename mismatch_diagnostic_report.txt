DIAGNOSTIC REPORT - PRE-FIX
==========================

1. FUND BALANCES VS TRANSACTIONS
--------------------------------
Findings:
- `fund_transfer` logic in `process_transaction` (migration 20260324191505) creates only ONE transaction (type='out' for the source fund).
- It updates the target fund balance directly: `UPDATE funds SET balance = balance + p_amount WHERE id = p_to_fund_id;`
- BUT, the `recalculate_fund_balance` function (migration 20260420064736) calculates balance ONLY from the `transactions` table.
- RESULT: Every time a recalculation trigger runs, the target fund LOSES the transferred amount because there is no corresponding 'in' transaction.
- MISMATCH: High probability of drift for all funds involved in transfers.

2. CONTACT BALANCES VS TRANSACTIONS
-----------------------------------
Findings:
- `sync_contact_balances` is often called manually from the frontend.
- `v_contact_balance` view correctly sums transactions.
- Risk: If manual sync is missed or fails, `contacts.balance` stays stale.
- `get_dashboard_snapshot` (migration 20260422062858) uses `contacts` table for `ledger_debit` and `ledger_credit` instead of summing transactions directly.

3. DASHBOARD TOTALS VS LEDGER TOTALS
------------------------------------
Findings:
- `get_dashboard_snapshot` calculates `netCompanyProfit` as `v_total_in - v_total_out` from `transactions`.
- However, `v_total_in` and `v_total_out` EXCLUDE `category != 'fund_transfer'`.
- Since transfers only have an 'out' part, this might actually be incorrectly deducting transfers from profit if not careful (though the `!= 'fund_transfer'` filter attempts to avoid this).
- The main issue is inconsistency between different parts of the UI that might use `calculationEngine.ts` (React-side) vs the RPC.

4. PERFORMANCE - useTransactions
--------------------------------
- PAGE_SIZE = 1000 causes significant lag.
- No server-side filtering means the full payload is always sent.
- Realtime triggers a full refetch of 1000 items on every small change.

5. TYPOS & DEBT
---------------
- `isManullyClosed` exists in 4 files.
- `ledger_accounts` table is still being referenced in `restore-backup`.
